# 通用 Java AI 开发规范

本规范用于约束 AI 在 Java 项目中的分析、设计、编码、测试、文档与交付行为。

参考来源：
- https://github.com/piomin/claude-ai-spring-boot/tree/main
- https://github.com/JetBrains/junie-guidelines/tree/main

这不是某个具体项目的脚手架约束，而是一份适用于大多数 Java / Spring Boot 企业项目的通用开发规范。目标是让 AI 产出的代码具备以下特征：
- 一致：命名、结构、接口风格、异常语义一致。
- 简洁：优先最小充分实现，避免无谓抽象。
- 可维护：边界清晰、职责单一、易于演进。
- 可验证：测试充分、行为明确、结果可证明。
- 工程化：适合真实项目长期维护，而不是一次性示例代码。

## 1. AI 工作原则

### 1.1 先理解，再动手
- 在实现前，先明确需求目标、上下文、输入输出、约束条件和验收方式。
- 如果用户要求与现有代码结构冲突，优先分析影响范围，再决定改法。
- 任何实现都必须围绕“当前目标”展开，不为了显得完整而额外扩展需求。

### 1.2 非简单任务必须先规划
- 对任何非琐碎任务，先给出计划，再进入实现。
- 满足以下任一条件时，必须先规划：
  - 包含 3 个及以上明显步骤；
  - 涉及多个模块或跨层改动；
  - 涉及数据模型、接口协议、事务边界、并发模型或部署方式变更；
  - 需要补充或调整测试策略。
- 计划至少应覆盖：
  - 改什么；
  - 为什么这样改；
  - 风险点；
  - 如何验证。

### 1.3 先收敛设计，再开始编码
- 在编码前，先确定关键接口、领域对象、异常模型、事务边界和测试范围。
- 当存在多种实现路径时，优先选择：
  - 更简单；
  - 更清晰；
  - 更少副作用；
  - 更容易测试和维护的方案。
- 避免边写边改方向，导致结构不断膨胀。

### 1.4 用户反馈应沉淀为长期约束
- 用户一旦纠正命名、结构、实现习惯、交付风格或工程约束，后续任务应默认继承。
- 对重复出现的问题，应主动形成稳定规则，而不是被动等待再次指出。

### 1.5 完成前必须验证
- 未经验证，不得声称任务完成。
- 交付前必须确认：
  - 代码行为符合目标；
  - 没有引入明显回归风险；
  - 测试覆盖主要正向/反向场景；
  - 文档、配置、依赖是否需要同步更新。
- 若条件允许，应运行自动化测试；若无法运行，必须说明原因和建议验证方式。

### 1.6 优先优雅、但反对过度设计
- 优先根因修复，而不是表面补丁。
- 优先最小改动面，但不能以牺牲设计清晰度为代价。
- 简单问题不做复杂设计；复杂问题也不能用临时拼接方式掩盖。
- 当一个实现“能跑但别扭”时，应主动考虑是否存在更自然、更稳定的结构。

## 2. 通用工程原则

### 2.1 简洁优先
- 优先写少而清楚的代码，而不是写多看起来“全面”的代码。
- 不要提前实现未来可能需要、但当前并未要求的能力。
- 不要引入无业务价值的抽象层、基类、工厂、模板体系或配置开关。

### 2.2 清晰优先于聪明
- 可读性优先于技巧性。
- 不要为了“高级写法”牺牲可理解性。
- 代码应让团队成员在不依赖口头解释的情况下读懂。

### 2.3 一致性优先
- 同一项目中，同类问题应使用同类解决方案。
- 命名风格、DTO 组织方式、异常风格、日志格式、测试结构应保持一致。
- 新代码要服从现有项目主风格，除非明确在做规范化重构。

### 2.4 面向变更设计
- 关键边界要清晰，以便后续演进。
- 不把数据库结构、第三方协议或临时实现细节泄漏到整个系统。
- 对外暴露的契约应稳定，对内实现应允许替换。

### 2.5 生产可用性思维
- 代码不仅要“功能正确”，还要考虑：
  - 配置是否可控；
  - 错误是否可定位；
  - 日志是否足够；
  - 行为是否可测试；
  - 是否容易部署与监控。

## 3. 项目结构规范

### 3.1 模块组织
- 按业务边界或清晰技术边界组织代码，不按“工具类堆积”组织项目。
- 避免所有代码都堆在单一包下。
- 若项目规模较大，可按模块划分为：
  - API / Web 层；
  - Application / Service 层；
  - Domain 层；
  - Infrastructure 层。

### 3.2 包结构
- 包名全部小写。
- 包名应短、稳定、有业务语义。
- 避免使用含糊包名，如 `common`, `utils`, `misc`, `temp` 作为主要承载层。
- 工具类若不可避免，应限制作用范围，避免演变成“全局垃圾桶”。

### 3.3 文件职责
- 一个类只承担一种主要职责。
- 一个文件中不要塞入多个无强关联的顶级概念。
- 当一个类开始同时负责校验、转换、存储、编排、日志控制时，应考虑拆分。

### 3.4 DDD 目录约定
- 微服务默认采用 `api / application / domain / infrastructure` 四层结构。
- `api` 只负责协议适配，不放核心业务决策。
- `application` 负责用例编排、事务边界、领域对象装配。
- `domain` 负责业务语义、聚合、值对象、领域仓储接口、领域规则。
- `infrastructure` 负责数据库、消息、缓存、第三方网关等技术实现。
- 数据库访问默认放在 `infrastructure.persistence` 下，按 `dataobject / mapper / repository` 细分。
- 领域对象不要直接承担持久化注解职责；数据库注解应放在 DO 上。
- 新增微服务骨架时，至少应具备：
  - `domain.model`
  - `domain.repository`
  - `application`
  - `infrastructure.config`
  - `infrastructure.persistence.dataobject`
  - `infrastructure.persistence.mapper`
  - `infrastructure.persistence.repository`

## 4. Java 语言规范

### 4.1 命名约定
- 包名使用全小写。
- 类名、接口名、枚举名使用 `PascalCase`。
- 方法名、变量名使用 `camelCase`。
- 常量使用 `UPPER_SNAKE_CASE`。
- 布尔变量命名优先体现语义，如 `enabled`, `active`, `deleted`，而不是模糊缩写。
- 方法命名应体现动作或语义，例如 `createOrder`, `loadAccount`, `isExpired`。
- 避免无意义缩写，如 `obj`, `data`, `tmp`, `info`，除非语境非常明确。

### 4.2 代码格式
- 使用 4 个空格缩进，不使用 Tab。
- 单行尽量不超过 120 个字符。
- 链式调用、长参数列表、多条件判断应适当换行，提高可读性。
- 同一代码块中保持统一格式风格，不混用不同排版习惯。
- 在业务规则不直观、状态流转复杂、事件语义容易误解、基础设施装配不明显的地方，应补充简洁中文注释。
- 注释要解释“为什么这样设计”或“这段代码在做什么业务动作”，不要写无信息量的废话注释。
- 简单 getter、显而易见的 record、直白赋值不需要强行加注释，避免注释噪音。

### 4.3 类设计
- 类应小而聚焦。
- 优先不可变对象，尤其是值对象、DTO、命令对象、事件对象。
- 纯数据承载优先使用 `record`，前提是确实适合不可变建模。
- 优先组合而不是继承，除非存在明确稳定的 is-a 关系。
- 继承层级不应过深。

### 4.4 接口设计
- 面向接口编程，但不要为了形式主义给每个类都创建接口。
- 当某个能力确实存在多实现、测试替身需求或边界抽象价值时，再引入接口。
- 接口粒度应适中，避免既过大又过碎。

### 4.5 枚举与常量
- 用 `enum` 替代业务状态上的字符串常量或数字常量。
- 避免魔法值散落在代码中。
- 常量只应抽取真正会复用、且具备清晰语义的内容。

### 4.6 空值与 Optional
- 返回值可能为空时，优先用 `Optional` 表达“可能不存在”。
- 不要把 `Optional` 用作字段类型。
- 不要把 `Optional` 用作方法参数。
- 不要把 `Optional` 当作一切空值问题的替代品；在集合场景下，优先返回空集合。

### 4.7 集合使用
- 根据语义选择 `List`、`Set`、`Map`。
- 返回集合时返回空集合，不返回 `null`。
- 判断集合为空时使用 `isEmpty()`。
- 需要顺序时明确使用保序集合。
- 不要在不必要的场景暴露可变集合。

### 4.8 Stream 使用
- Stream 中避免副作用。
- 逻辑过于复杂时，不要强行使用 Stream。
- 能用方法引用时，优先于冗长 lambda。
- 避免在 Stream 流水线中塞入复杂分支，导致调试困难。

### 4.9 异常处理
- 只捕获真正能够处理的异常。
- 不要直接吞掉异常。
- 不要滥用顶层 `RuntimeException` 包装一切。
- 业务异常、资源不存在、参数非法、外部依赖错误等应在语义上区分。
- 抛出的异常信息应帮助定位问题，但不能泄露敏感数据。

### 4.10 日期时间
- 统一优先使用 `java.time.*`。
- 明确区分日期、本地时间、带时区时间和时间戳语义。
- 跨系统交互时，对时区和序列化格式保持明确约定。

### 4.11 字符串与文本
- 多行文本优先使用 Text Block。
- 不要在循环中做低效字符串拼接。
- 当字符串具有结构语义（例如 SQL、JSON、模板片段）时，应优先保证可读性。

### 4.12 并发
- 优先使用 `java.util.concurrent` 下的高级并发工具。
- 仅在简单原子可见性语义下使用 `volatile`。
- 并发代码必须明确共享状态、锁粒度、线程安全边界。
- 对并发代码，优先选择更简单、可验证的实现方式。

## 5. 面向对象与领域建模

### 5.1 单一职责
- 一个类只应围绕一个主要变化原因演化。
- 当类中开始出现明显不属于同一职责的字段和方法时，应拆分。

### 5.2 领域语义优先
- 领域对象命名应体现业务含义，而不是技术实现细节。
- 服务方法命名应表达用例，而不是底层操作顺序。
- 不要把所有逻辑都推到 `ServiceImpl` 中成为“大泥球”。

### 5.3 值对象与实体分离
- 有明确身份标识、生命周期和持久化含义的对象可建模为实体。
- 只表达属性组合、不强调身份的对象可建模为值对象。
- 不要为简单场景过度引入 DDD 术语，但对象语义要尽量清晰。

### 5.4 命令与查询分离
- 写操作的输入建议建模为命令对象。
- 读操作的筛选条件、分页条件建议建模为查询对象或明确参数对象。
- 不要把增删改查全部塞进一个“万能请求对象”。

## 6. Spring Boot 规范

### 6.1 依赖注入
- 统一优先构造器注入。
- 必需依赖使用 `final` 字段声明。
- 单构造器场景下不必显式写 `@Autowired`。
- 避免字段注入和 Setter 注入用于生产代码。

### 6.2 组件可见性
- Spring 组件、配置类、`@Bean` 方法、Controller 及其处理方法，在可行时优先使用包级可见性。
- 只暴露真正需要被其他包使用的 API。

### 6.3 配置管理
- 配置项按业务域分组，使用统一前缀。
- 优先使用 `@ConfigurationProperties` 管理应用配置。
- 配置类要加校验，尽早失败。
- 不要把同一类配置散落在多个 `@Value` 注入点中。
- 环境差异优先通过环境变量传递，而不是依赖复杂 profile 组合。

### 6.4 生命周期与初始化
- 初始化逻辑应明确、可控、可测试。
- 避免在 Bean 初始化阶段做过重、不可控、强耦合的远程调用。
- 对启动期失败要有明确策略，避免“半可用”状态。

### 6.5 事务边界
- Service 层方法应明确表达事务边界。
- 查询使用 `@Transactional(readOnly = true)`。
- 写操作使用 `@Transactional`。
- 尽量缩小事务范围，避免事务中掺杂无关远程调用或重计算逻辑。

### 6.6 Web 层与持久化层分离
- 不要直接暴露 JPA Entity 作为接口请求或响应。
- Web 层使用明确的请求 DTO、响应 DTO。
- Persistence 模型、Domain 模型、API 模型职责应清晰。
- DDD 下领域对象不直接承担数据库注解职责，数据库映射应优先放在 `infrastructure.persistence` 下的 DO / Mapper 中。
- 若映射逻辑开始重复，可引入稳定的映射层；不要一开始就过度抽象。

### 6.7 MyBatis-Plus 约定
- 默认使用 MyBatis-Plus 作为 ORM / 持久化增强层。
- Mapper 接口统一放在 `infrastructure.persistence.mapper` 包下。
- DO 统一继承共享基础 DO，复用 `id`、`version`、`createdAt`、`updatedAt` 等字段。
- 分页能力、自动填充、乐观锁等公共能力统一放在共享持久化模块中配置。
- 复杂 SQL 优先写到 Mapper XML；简单单表操作优先复用 MyBatis-Plus 基础能力。
- 不要把业务规则写在 Mapper 里，Mapper 只负责数据访问。

### 6.8 Druid 约定
- 默认连接池使用 Druid。
- 数据源配置统一写在 `spring.datasource.druid` 下。
- 连接地址、用户名、密码、池大小、校验 SQL 必须显式配置，不依赖隐式默认值。
- 新增服务时，端口和数据库连接必须在各自 `application.yml` 中单独配置。

### 6.9 REST API 设计
- 接口路径以资源为中心，避免动词式混乱命名。
- 推荐使用版本化路径，如 `/api/v1/orders`。
- 状态码必须与行为匹配：
  - 成功读取返回 `200`；
  - 成功创建返回 `201`；
  - 无内容删除可返回 `204`；
  - 参数错误返回 `400`；
  - 未找到返回 `404`；
  - 冲突返回 `409`；
  - 服务内部错误返回 `500`。
- 集合资源应支持分页、排序或筛选（如场景需要）。
- JSON 顶层优先用对象结构，方便未来扩展元数据。
- 字段命名风格在系统内保持统一。

### 6.8 参数校验
- 输入校验前置到边界层。
- 请求 DTO 使用 Jakarta Validation 注解。
- 对校验失败返回一致的错误结构。
- 不要把大量 if/else 参数检查散落在 Service 中。

### 6.9 统一异常处理
- 使用 `@RestControllerAdvice` 或 `@ControllerAdvice` 集中处理异常。
- 错误响应结构统一、可预测。
- 错误响应至少应包含：
  - 错误类型或错误码；
  - 用户可理解的信息；
  - 机器可消费的标识；
  - 时间或上下文信息（按项目需要）。
- 可优先参考 Problem Details 模型。

### 6.10 日志
- 使用 SLF4J 与标准日志后端。
- 严禁使用 `System.out.println()` 输出业务日志。
- 不记录密码、令牌、密钥、证件号、手机号、邮箱等敏感数据，除非明确做过脱敏。
- 高成本 DEBUG/TRACE 日志使用级别判断或延迟求值。
- 日志信息要能够帮助定位问题，但避免重复和噪音。

### 6.11 监控与运维友好性
- 仅暴露必要 Actuator 端点。
- 其他敏感端点必须受保护。
- 健康检查、指标、版本信息命名应稳定。
- 代码应考虑容器化运行、配置注入和外部化部署场景。

### 6.12 国际化
- 面向用户可见的消息、提示、错误文案外置到 ResourceBundle。
- 不要在代码中到处硬编码可见文本。

## 7. 数据访问规范

### 7.1 持久化边界
- Repository 负责数据访问，不承担复杂业务编排。
- Service 负责用例编排、事务与业务规则。
- 不要把业务逻辑下沉到 Controller，也不要把 HTTP 语义塞进 Repository。
- 不同微服务使用独立数据库时，表名仍统一加上微服务名前缀，避免跨库迁移、汇总分析、对账脚本或后续合库时出现同名表冲突。

### 7.1.1 表命名约定
- 表名使用小写下划线风格。
- 表名前必须带微服务名前缀，格式为：`<service_prefix>_<table_name>`。
- 前缀使用业务服务短名，而不是完整 artifact 名；例如：
  - `user-service` 使用 `user_`
  - `account-service` 使用 `account_`
  - `order-service` 使用 `order_`
  - `market-service` 使用 `market_`
  - `futures-service` 使用 `futures_`
- 关联表、流水表、事件表、配置表同样遵守此前缀规则，不能例外。

### 7.2 MyBatis-Plus 使用建议
- 使用 MyBatis-Plus 时，SQL、Mapper、DO 与领域对象职责分离，不要把领域逻辑塞进 Mapper。
- 明确控制查询字段、分页范围和批量操作边界，避免隐式全表扫描。
- 复杂查询优先写清楚 SQL 语义，不要为了通用封装牺牲可维护性。

### 7.3 查询设计
- Repository 方法命名要清楚表达意图。
- 复杂查询可使用显式查询定义，但不要牺牲可读性。
- 只查询真正需要的数据。
- 大结果集要分页，避免一次性全量读取。

### 7.4 数据一致性
- 更新逻辑要明确并发与幂等语义。
- 对唯一约束、重复提交、状态竞争等场景要有清晰策略。
- 事务一致性需求必须在设计期明确，而不是依赖偶然行为。

## 8. 安全与稳健性规范

### 8.1 输入永远不可信
- 所有外部输入都应视为不可信，包括 HTTP 请求、MQ 消息、数据库外部导入、配置项和第三方回包。
- 在边界层做校验、约束和格式化。

### 8.2 最小暴露原则
- 只暴露必要接口、字段和错误信息。
- 不要把内部实现细节直接透出到 API 或日志中。

### 8.3 敏感信息保护
- 机密信息应通过环境变量或安全配置传入。
- 不要把密钥、账号、令牌写死在代码或测试样例中。
- 日志、异常、监控标签中避免泄露敏感数据。

### 8.4 失败要可控
- 外部依赖调用失败时，要有明确异常语义。
- 超时、重试、降级、幂等等机制是否需要，应由需求驱动，而不是默认乱加。
- 不要在用户没有要求时随意加入兜底逻辑，避免掩盖真实问题。

## 9. 测试规范

### 9.1 测试是交付的一部分
- 非演示代码必须有测试。
- 测试不是附属品，而是交付正确性的证明。

### 9.2 测试覆盖范围
- 至少覆盖：
  - 正常路径；
  - 参数非法与边界条件；
  - 关键异常路径；
  - 关键状态变化；
  - 序列化/反序列化、校验、事务或持久化行为（如适用）。

### 9.3 单元测试
- 聚焦单个类或单个行为。
- 测试命名应清楚表达场景与期望。
- 避免过度 mock 导致测试只验证“mock 被调用”。
- 单元测试要快、稳定、可重复。

### 9.4 集成测试
- 验证真实 Spring 上下文、数据库、消息系统、HTTP 行为等集成效果。
- 优先使用 Testcontainers 启动真实依赖。
- 容器镜像使用明确版本，不使用 `latest`。
- Spring Boot 集成测试优先使用随机端口：

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
```

### 9.5 测试数据
- 测试数据应最小、清晰、与场景强相关。
- 不要堆砌大量无意义测试数据。
- 测试数据命名应能表达业务意图。

### 9.6 完成标准
- 相关测试通过后，才能认为实现完成。
- 若本地环境无法运行测试，必须在交付中明确指出：
  - 哪些测试未执行；
  - 为什么未执行；
  - 建议如何补验。

## 10. 构建、依赖与发布规范

### 10.1 构建工具
- Java 项目优先使用 Maven 或 Gradle，并保持项目内一致。
- 不要同时混用两套构建体系作为主入口。

### 10.2 依赖管理
- 依赖版本应明确、稳定、可维护。
- 优先选择成熟、主流、社区验证充分的库。
- 不要为了一个小功能引入重量级依赖。
- 新增依赖前，先评估是否已有现成能力可复用。

### 10.3 版本管理
- 对外发布的项目建议遵循语义化版本管理。
- 破坏性变更、功能新增、缺陷修复应在版本上有清晰体现。

### 10.4 本地运行与基础设施
- 若项目依赖数据库、缓存、消息系统或其他基础设施，应提供本地运行方式。
- 推荐通过 `docker-compose.yml` 或等效方式统一启动依赖。
- README 应写清楚启动、构建、测试、运行步骤。

### 10.5 CI 要求
- 项目应具备最基本 CI 校验：编译、测试、必要静态检查。
- 生成或初始化项目时，应同步考虑 CI 配置，而不是把验证完全留给人工。

## 11. 文档规范

### 11.1 文档必须随代码演进
- 新增模块、配置、接口、依赖、运行方式时，应同步更新文档。
- 文档滞后会直接降低团队可维护性。

### 11.2 README 至少应说明
- 项目用途；
- 技术栈；
- 启动方式；
- 配置方式；
- 测试方式；
- 外部依赖；
- 关键接口或模块说明（按项目需要）。

### 11.3 面向使用者写文档
- 文档应说明“怎么用”和“为什么这样设计”。
- 避免只写口号式描述，不写关键步骤。

## 12. AI 输出约束

### 12.1 输出必须可落地
- 优先给出可编译、可运行、可验证的实现。
- 避免给出只有示意意义、无法接入现有工程的伪代码。

### 12.2 不擅自添加未要求能力
- 不主动加入用户未要求的 fallback、降级、备用数据源或兜底逻辑。
- 不擅自改变用户指定的数据来源、协议或边界条件。
- 不为了“保险”而修改原始需求语义。

### 12.3 解释要聚焦设计原因
- 给出方案时，应能说明：
  - 为什么这样组织；
  - 关键边界在哪里；
  - 如何验证；
  - 后续如何维护。

### 12.4 改动要最小充分
- 只改与任务直接相关的内容。
- 避免顺手大改无关模块，除非它们阻塞当前任务。
- 如果发现更大范围问题，先说明，再决定是否扩展处理。

### 12.5 每次改代码的强制流程
AI 每次修改代码时，必须按以下流程执行并在交付中体现：
1. 确定需求：先确认本次要解决的问题、输入输出、约束条件和验收标准，不明确时先收敛需求再编码。
2. 查看现有逻辑：阅读与需求直接相关的代码、配置、测试和文档，理解当前实现方式和项目既有风格。
3. 评估影响范围：明确会影响的模块、接口、数据模型、配置、测试、文档和潜在回归点。
4. 执行最小修改：只改必要代码；完整实现新增功能；不得影响现有功能；需要复用时做合理的方法抽取和逻辑抽象，避免无意义封装。
5. Review 代码：完成后自查命名、边界、异常、事务、日志、可读性、兼容性和是否存在过度设计。
6. 新增功能测试和回归测试：为新增功能补充或更新测试；回归测试必须覆盖改动代码直接涉及的现有功能；无法运行时必须说明原因和建议补验方式。
7. 输出本次改动：交付时说明改了哪些文件、实现了什么、影响范围、测试结果和未完成/需注意事项。

## 13. 推荐执行流程

AI 在 Java 项目中默认遵循以下流程：
1. 理解需求、上下文与约束。
2. 判断是否需要先规划。
3. 收敛接口、模型、事务、异常与测试策略。
4. 做最小必要实现。
5. 补齐或更新测试。
6. 运行验证。
7. 更新 README 或相关文档。
8. 交付时说明改动内容、原因、影响范围与验证方式。

## 14. 适用范围说明

本规范适用于：
- 通用 Java 后端项目；
- Spring Boot 微服务或单体服务；
- 需要 AI 参与编码、重构、补测试、补文档的团队项目。

本规范不强制要求：
- 必须采用某一种架构流派；
- 必须同时引入 DDD、CQRS、事件驱动等整套复杂模式；
- 必须为了规范而牺牲项目现有上下文。

最终原则只有一个：
在满足用户目标和现有项目约束的前提下，产出简洁、稳定、可维护、可验证的 Java 代码。
